## Current Baseline

The harness now has the core safe coding-agent loop:

- CLI chat shell with styled output and interactive menus.
- Task state under `.agent/tasks`.
- Event, command, review, belief, checkpoint, and apply-back artifacts.
- File listing, reading, search, patch, write, command, and check tools.
- Permission boundaries for risky operations.
- Tracked undo.
- Shadow workspace execution with apply-back and conflict detection.
- OpenRouter tool-calling adapter.
- Final review card and critique pass.
- Structured check parsing and bounded repair loop.
- Targeted check runner with package-manager-aware command generation.
- Project-level memory with stale-source refresh and model-context integration.
- Explicit phase controller with per-task `phases.json`, phase events, artifact gates,
  and model tool filtering by phase.
- Apply-back conflict resolution with per-file keep/apply/save choices and conflict
  artifacts.
- Better approval/review actions with richer menus, typed fallbacks, technical details,
  changed-file list shortcuts, alternate-approach approval denial, and task cancellation.
- Model adapter contract, provider capability flags, transient retry, fallback model
  configuration, structured critique output, streaming text hook, and per-task usage/cost
  tracking.
- Better diff/review UX with changed-file reasons, blast radius, check snippets,
  severity-grouped warnings, and task timeline.

Phase 2 focused on agent capability: helping the model find the right code,
understand failures, iterate safely, and deliver working changes more often.

## What Is Left

No Phase 2 items remain. The next session should start by creating a Phase 3
roadmap before implementing more features.

Recommended Phase 3 candidates:

1. Real language-server or deeper code intelligence: TypeScript/Python LSP support,
   cross-file binding resolution, dependency graph, route maps, and component maps.
2. Stronger targeted checks and parsers: framework-specific failure parsing, build
   parser depth, and better failed-test-to-source mapping.
3. Provider/model polish: wire streaming into the CLI, add broader provider adapters
   beyond OpenRouter, and use structured outputs beyond critique.
4. Productized task/session controls: stronger cancel-in-flight behavior,
   alternate-approach flow, and resumable tasks.
5. Safer multi-file/codebase operations: pre-patch blast radius, richer
   merge/conflict strategy, and dependency-aware edits.
6. GitHub/CI integration: PR creation, check status inspection, CI log repair, and
   branch workflows.
7. Hardening against real repos: fixture suites, end-to-end CLI tests, and more
   failure-mode coverage.

## 2. Targeted Check Runner

Status: complete. Reviewed and hardened.

### Goal

Avoid running only broad package scripts. Let the agent run focused checks when it knows the affected file or test.

### Add

- [x] `detect_test_runner` — detects Jest, Vitest, Mocha, Playwright, Cypress, Node built-in, pytest, Cargo, Go
- [x] `run_test_file(path)` — runs a specific test file using the detected runner
- [x] `run_test_name(name)` — runs a test by name pattern using the detected runner
- [x] `run_related_tests(changed_file)` — finds co-located test files and runs them
- [x] `run_typecheck` basic package-script wrapper
- [x] `run_lint` basic package-script wrapper
- [x] `run_build` basic package-script wrapper

Runner detection covers: Vitest, Jest, Playwright, Cypress, Mocha, Node built-in test runner, pytest, Cargo, Go.

Related-test discovery uses co-location conventions: `.test.js`, `.spec.ts`, `test/<name>.test.js`, `__tests__/`, etc.

The repair loop now prefers targeted test runs when it knows which files failed.

Model tools exposed: `detect_test_runner`, `run_test_file`, `run_test_name`, `run_related_tests`.

### Hardening fixes applied

- `LOCAL_CHECK_RE` in `src/permissions/permission-engine.js` expanded to cover all targeted
  runner command forms (`npx jest`, `pnpm exec vitest`, `yarn mocha`, `bunx jest`,
  `python -m pytest`, `cargo test`, `go test`). Without this, every targeted run prompted
  for approval regardless of `allowLocalChecks`.
- `detectTestRunner` now reads lock files and emits the correct package-manager-aware exec
  prefix (`npx`, `pnpm exec`, `yarn`, `bunx`). Returned descriptor now includes `packageManager`.

### Definition Of Done

If a patch only touches `src/foo.test.ts`, the agent can run that test file instead of the full suite when supported.

## 3. Code Intelligence Tools

Status: complete (lightweight MVP).

### Goal

Help the agent navigate real codebases without relying only on text search.

### Add

- [x] `find_symbols(query)`
- [x] `find_definition(symbol_or_identifier)`
- [x] `find_references(symbol_or_identifier)`
- [x] `find_imports(path)`
- [x] `find_exports(path)`
- [~] `dependency_graph(path?)` — implicit via `find_imports`; no aggregated tool yet
- [x] `route_map()`
- [ ] `component_map()` — not built; React component discovery still falls back to symbol/text search

### MVP Approach

Implemented with regex-based parsers, no new dependencies:

- TypeScript/JavaScript: ESM imports/exports, CommonJS `require` and `module.exports`,
  function/class/interface/type/enum/variable detection. Routes covered:
  Express/Fastify/Koa/Hono, React Router, Next.js `pages/` and `app/` file routes.
- Python: `def`/`class` symbols and `import` / `from x import y` statements.
- Index lives in `src/code/code-index.js`, cached per runtime, invalidated on edits.

Language-server integration can come later.

### Definition Of Done

The agent can answer:

```txt
Where is the dashboard route rendered?
Who calls updateUser?
Which tests mention login redirect?
```

using structured evidence.
